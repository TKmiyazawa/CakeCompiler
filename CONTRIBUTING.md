## 📋 Issue（相談）の書き方

「ここがおかしい」「こうしたらもっといいのに」と思ったら、**Issue（イシュー）** を書いてみましょう。Issue とは、GitHub 上での「相談メモ」のようなものです。

### ステップ

1. [Issues ページ](https://github.com/TKmiyazawa/CakeCompiler/issues) を開く
2. 右上の緑色のボタン **「New issue」** をクリックする
3. タイトルと本文を書く

### 書き方のコツ

タイトルは短く、本文はできるだけ具体的に書くのがコツです。完璧（かんぺき）じゃなくて大丈夫。伝わればOKです。

**バグ（不具合）の報告：**

```
タイトル：ケーキを選んだあと画面が固まる

何をしたか：フルーツタルトのカードをタップして「このケーキに決める」を押した
何が起きたか：画面が真っ白になって動かなくなった
使っている端末：Pixel 7a / Android 14
```

**アイデアの提案：**

```
タイトル：新しいケーキ「カヌレ」を追加したい

カヌレは外がカリカリ、中がもっちりしていて、
甘さ（sweetness）は 0.6、食感（texture）は 0.9 くらいだと思います！
```

**分からないことの質問：**

```
タイトル：PreferenceVector の数字の意味が分からない

0.7 とか 0.3 って何を基準にしているんですか？
README を読んだけど、もう少し詳しく知りたいです。
```

> 💡 **ヒント：** 「こんなこと聞いていいのかな？」と思ったら、それこそ聞いてください。同じことで悩んでいる人が、きっと他にもいます。

## 🔧 Pull Request（提案）の基本

コードを直してみたい、新しい機能を追加してみたい。そう思ったら、**Pull Request（プルリクエスト、略して PR）** に挑戦してみましょう。

PR とは、「こういう変更を提案します。見てもらえますか？」というリクエストのことです。

### はじめての PR の手順

1. **このリポジトリを Fork（フォーク）する**
   - 右上の「Fork」ボタンを押すと、自分のアカウントにコピーが作られます

2. **自分のコピーをパソコンにダウンロードする**
   ```bash
   git clone https://github.com/あなたのユーザー名/CakeCompiler.git
   ```

3. **新しいブランチ（作業場所）を作る**
   ```bash
   git checkout -b add-canele-cake
   ```

4. **コードを編集する**

5. **変更を保存して送る**
   ```bash
   git add .
   git commit -m "新しいケーキ「カヌレ」を追加"
   git push origin add-canele-cake
   ```

6. **GitHub で Pull Request を作成する**
   - 自分のリポジトリのページに「Compare & pull request」というボタンが表示されるので、クリックします

### 心配しなくていいこと

- **完璧じゃなくていい。** PR を出したあとに「ここをこう直すともっといいよ」とアドバイスがもらえます。それが PR の良いところです。
- **壊す心配はない。** 君の PR は、元のコードに直接反映されるわけではありません。レビュー（確認）を経て、初めてマージ（統合）されます。
- **小さい変更でいい。** コメントの誤字を直す、ケーキの説明文を1行加える。それだけで立派な PR です。

> **最初の一歩が一番難しい。でも、一歩踏み出せば、そこはもう「開発者」の世界です。**

## 🏠 コミュニティ・ルール

このプロジェクトに参加するすべての人に、以下のルールを守ってもらいたいと思っています。

### 1. お互いを尊重（そんちょう）する

プログラミング歴が10年の人も、今日はじめて GitHub を開いた人も、ここでは同じ仲間です。相手の言葉に敬意（けいい）を持って接してください。

### 2. わからないことを恥（は）ずかしがらない

「こんな初歩的（しょほてき）なこと聞いていいのかな？」——いいんです。質問すること自体が、学びの第一歩です。誰だって最初は初心者でした。

### 3. 失敗を歓迎（かんげい）する

コードが動かない？ PR がうまくいかない？ 大丈夫。**失敗は、学んでいる証拠（しょうこ）です。** このプロジェクトでは、失敗を笑う人はいません。

### 4. 建設的（けんせつてき）な言葉を選ぶ

誰かのコードや提案に対して意見を言うときは、「ここがダメ」ではなく「こうするともっと良くなるかも」という言い方を心がけてください。
